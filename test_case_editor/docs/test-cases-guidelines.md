# Требования к оформлению тест-кейсов

## Название
- Емкое, отражает суть проверки
- Без лишних слов, эпитетов, номеров спецификаций
- При длинном заголовке требуется декомпозиция

## Общие требования
- Без противоречий и двусмысленностей
- Исключить «и т. п.», «и т. д.»
- Избегать избыточных деталей интерфейса
- Понятен без знания проекта
- Воспроизводим без консультации автора
- Пригоден для автоматизации
- Без дублирующих сценариев
- Включает позитивные и негативные проверки

## Структура и оформление
- Один абзац — одна мысль/действие
- Шаги: ответ на «Что сделать»
- Результаты: ответ на «Что делает»
- Все шаги атомарны и проверяемы
- Без однотипных повторяющихся формулировок
- Списки: новая строка + дефис
- Предложения с заглавной буквы, точкой в конце
- Исключить деепричастные обороты

## Содержание
- Тестовые данные, вложения, учетные записи
- Расшифровка терминов и ролей пользователей
- Шаги: от входа до итогового результата
- Исходные и выходные данные, файлы
- Для нагрузочных сценариев: объемы данных и бенчмарки
- Время выполнения набора: 15–120 минут

## Ревью тест-кейсов
- Проверять понятность и читаемость
- Следить за соответствием методике
- Проверять полноту покрытия (все критерии приемки)
- Отмечать орфографические ошибки
- Исключать инлайн-скрины
- Повторяющиеся ошибки одного типа отмечать один раз
- Советы дружественно — коллега к коллеге
- Помогать делать кейсы лаконичными и удобными
- Проверять корректность входных/выходных данных и последовательность шагов
